## 配布形態に関する参考情報

「社内利用のみ」の場合、一般的にはソフトウェアの「配布」に該当しないと考えられるため、コピーレフト条項（GPL等）の適用範囲は限定的になるという見解があります。ただし、組織の法務方針によって解釈が異なる場合があるため、RED判定のライセンスが含まれる場合は確認が考えられます。

## 判定詳細

### GREEN判定

**Apache-2.0ライセンス（#3, #4, #5, #6, #8, #9）**
- Spring Framework、Spring Boot、Reactor Netty、springdoc-openapiはすべてApache-2.0
- プロプライエタリプロジェクトとの互換性があるとされるライセンス
- 著作権表示およびNOTICEファイルの保持が求められる点を確認することが考えられます

**EPL-2.0 / GPL-2.0-with-classpath-exception（#1, #2）**
- Jakarta XML Web Services API および JAX-WS実装はデュアルライセンス
- 「Classpath Exception」付きのGPL-2.0は、ライブラリとしてリンクする場合にコピーレフト条項が適用されないという例外条項が付与されている形態です
- EPL-2.0を選択する場合も、プロプライエタリプロジェクトでの利用は一般的に可能とされています
- **デュアルライセンスのため、互換性の高い方（EPL-2.0またはClasspath Exception付きGPL-2.0）を選択できます**

### 対象外

**jp.co.createlink:core（#7）**
- 社内で開発されたComposite Buildライブラリのため、OSSライセンスチェックの対象外

## 推移的依存に関する注意

上記は直接依存のみの一覧です。推移的依存（依存の依存）については以下の観点で追加確認することが考えられます:

- Spring Boot BOMで管理される推移的依存は概ねApache-2.0ですが、一部例外がある可能性があります
- `jaxws-rt` の推移的依存にはJakarta関連ライブラリが含まれ、EPL-2.0/GPL-2.0-with-classpath-exceptionのものが含まれると考えられます
- より網羅的なチェックには、Gradleのライセンスプラグイン（例: `com.github.jk1.dependency-license-report`）の導入が考えられます

```bash
# 推移的依存を含むライセンス一括取得の例（プラグイン導入後）
./gradlew generateLicenseReport
```

## 推奨アクション

1. **著作権表示の確認**: Apache-2.0ライセンスの依存について、必要に応じてNOTICEファイルを成果物に含めることを検討
2. **推移的依存の網羅チェック**: Gradleライセンスレポートプラグインの導入による全依存の一括確認を検討
3. **Jakarta関連のライセンス選択**: デュアルライセンス（EPL-2.0 / GPL-2.0-with-classpath-exception）について、組織の方針に沿ったライセンス選択の明示を検討
